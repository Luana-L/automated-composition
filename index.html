Pitch Set Theory Composer

Type or click an initial pitch class sequence. Random T / I / R operations unfold the composition.

A pitch class set treats pitches as equivalent up to octave, so there are exactly twelve of them: C, C#, D, … B, laid out on the clock face below as 0–11. Pitch set theory gives you three transformations you can apply to any sequence of pitch classes:

  • Transpose (Tn) — add n mod 12 to every pitch class. On the clock, this is a rotation.
  • Invert (I) — reflect each pitch class across 0 (so 1↔11, 2↔10, …). On the clock, this is a mirror across the vertical axis.
  • Retrograde (R) — reverse the order of the sequence. The set doesn't change; only the reading order does.

The generator starts with your initial sequence and, at each step, randomly picks one operation and applies it to the current sequence — producing a chain of related but never-identical variations.

What I Learned

Pitch set theory

A pitch class is an integer mod 12; a pitch class sequence is a list of those integers; once you have a set of starting pitch classes, you can generate new sequences by transposing, inverting, or retrograding them:

transpose(seq, n) = seq.map(pc => (pc + n) % 12)
invert(seq) = seq.map(pc => (12 - pc) % 12)
retrograde(seq) = seq.slice().reverse()

Using this theory, you can build a generator that wanders through the family of related sequences defined by those operations. It's a different approach to automated composition than Markov chains or cellular automata — with only T, I, and R you'll only ever reach members of the same small, tightly constrained family of sequences. The flavor of the piece is fixed the moment you pick the input set. After that, the generator is just choosing which sibling to play next.

Implementation details

I play each segment two ways at once: an arpeggio reading the ordered sequence, and a soft pad holding the unordered set. The arpeggio is a throwaway OscillatorNode per note with a linear attack and exponential decay; the pad is a long-running oscillator per pitch class, each one routed through a StereoPannerNode so voices spread across the field. It gives a nice contrast between the operations that reorder the sequence (retrograde) and the ones that change it (transpose, invert), and the pad makes it easier to hear the underlying pitch set when the arpeggio wanders off.

A detail that surprised me was this line:

if (cur.op !== 'R' && !sameSet(prev.seq, cur.seq))
  refreshPadForCurrentSegment();

Retrograde only reorders the sequence, so the underlying set is identical — which means the old pad can just keep ringing while the arpeggio reverses over it. The algebra makes a UX decision for you: when two segments are pad-equivalent, don't interrupt the sustain. You hear exactly what R is.

For reverb I skipped shipping an impulse-response WAV and just synthesized one — a couple of seconds of exponentially decaying stereo noise written straight into an AudioBuffer and handed to a ConvolverNode. Not physically realistic, but it's short, has no fetch() race, and sounds like a modest room. The waveform dropdown is similarly simple: four of the options set osc.type to a built-in, the fifth ("rich") swaps in a custom PeriodicWave built from a small Fourier-coefficient array.

On a piano-roll the operations are abstract. On a 12-position clock they become geometry: rotation for T, reflection across the vertical axis for I, a direction flip for R. I draw the sequence as a polygon on a DPR-scaled canvas and watch that shape rotate or flip between segments. For symmetric sets like {0, 3, 6, 9} the polygon is a square, T3 leaves it invariant, and the piece goes still. For asymmetric sets the polygon rattles around and the piece feels restless. I didn't plan for the visual to be diagnostic of the music, but it is — and the decisions that made this sound like a piece instead of a demo weren't really in the generator at all, they were in the audio graph around it.
